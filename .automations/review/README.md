# Review Automation Records

这个目录用于记录“设计文档 vs 实现”的周期性审查结果。

## 目录约定

- `runs/`
  - 每次 automation 运行生成一个独立记录文件。
  - 文件名格式：`YYYY-MM-DDTHH-mm-ss.md`
- `summary.md`
  - 当前累计状态与未关闭问题摘要。
- `templates/run-template.md`
  - 单次审查记录模板。

## 审查范围

优先检查以下设计文档与实现是否一致：

- `docs/design/**`
- `design/**`
- `packages/*/DESIGN.md`
- 与以上设计直接对应的 `packages/**/src/**`、测试和 CLI/API 行为

## 记录规则

每次运行都必须：

1. 先阅读相关设计文档，再阅读对应实现。
2. 仅记录“设计明确表达过，但实现不一致、缺失、命名漂移、接口漂移或行为回退”的项。
3. 区分以下状态：
   - `open`: 已确认不一致，尚未修复
   - `resolved`: 之前存在，目前已对齐
   - `needs-clarification`: 设计文档本身不够明确，暂不能判定
4. 不把纯猜测写成结论；证据必须同时包含设计出处和实现出处。
5. 如果本轮没有发现问题，也要写明检查范围和结论。

## 严重级别

- `S0`: 核心架构或用户承诺明显不一致
- `S1`: 重要功能、接口或流程不一致
- `S2`: 命名、边界、默认行为或文档落后

## `summary.md` 维护规则

- 若文件不存在，则创建。
- 文件需保留：
  - 最近一次运行时间
  - 本轮检查范围
  - 当前 `open` 问题列表
  - 最近一次新增 / 已解决数量
- 已解决问题不要删除，移动到“Recently Resolved”区块，最多保留最近 20 条。
