# Control Review

Use this review when work is research-like, design/proposal space is wide, a
reframe is unsettled, or a major method is being imported. It is not a daily
workflow. Its job is to create useful opposition and synthesize back to one
mainline.

The user of this reference is the agent doing concrete engineering work:
software projects, product design, writing systems, research programs, or
project steering. Keep roles grounded in the work being advanced.

## Dialectical Work Review

Use this for ordinary attention-driven work: design, proposals, research,
reframe, writing structure, or project steering.

Run three independent roles:

- Advocate: why the proposal works and which load-bearing 30% it captures.
- Opposition: strongest objection, brittle assumption, or failure mode.
- Operator: what will happen when someone implements, uses, maintains, or
  continues this work.

Each role answers:

1. What is the central claim?
2. What assumption makes or breaks it?
3. What observation would change the judgment?
4. What smallest next move should follow?

Then synthesize:

```markdown

- Shared conclusion:
- Disagreements:
- Adopt now:
- Reject or defer:
- Smallest validation:
- Files to change:
```

Adopt only changes that improve local steering: clearer target, sharper system
shape, better observation, smaller correction, better disturbance handling, or
cleaner continuity for the next agent.

Do not preserve the full discussion unless it becomes load-bearing. A short
synthesis in a blueprint or record is usually enough.

## Method Import Review

Use this when importing an external theory or mature-domain method into the
current engineering work. The question is not "how do we imitate that method?"
but "how does this method help this project move?"

Run three independent roles:

- Source-domain expert: extracts the operating principle without copying form.
- Project engineer: maps the principle to the current system, constraints, and
  load-bearing 30%.
- Operator: checks whether the imported method can be used, maintained, and
  continued without becoming burden.

Each role answers:

1. What operating principle transfers?
2. What source-specific form should not be copied?
3. What local project problem does this actually improve?
4. What concrete change should be adopted, rejected, or tested?
